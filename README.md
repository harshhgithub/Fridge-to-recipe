# Fridge → Recipe

Paste in whatever ingredients you have at home, and get back a full recipe — scalable servings, a checklist of steps, and ingredient swaps — generated by an LLM and rendered as an interactive UI (not a chat box).

## What it does

- Type a free-form list of ingredients (e.g. "chicken breast, spinach, garlic, rice, half a lemon")
- The backend sends that to an LLM (Groq) with a system prompt that forces structured JSON output
- The frontend parses that JSON into an interactive recipe: adjustable servings (ingredient amounts scale live), a checklist of steps you can tick off, and a list of ingredient swaps

## Stack

- **Frontend:** React (hooks, functional components), Vite
- **Backend:** Express, calling the Groq API (`llama-3.3-70b-versatile`)
- **Icons:** lucide-react

## Setup

### 1. Backend

```
cd server
npm install
```

Create a `.env` file in `server/`:

```
GROQ_API_KEY=your_key_here
PORT=5000
```

Get a free Groq API key at [console.groq.com](https://console.groq.com).

```
npm start
```

Server runs on `http://localhost:5000`. Health check: `GET /api/health`.

### 2. Frontend

```
cd client
npm install
npm run dev
```

App runs on `http://localhost:5173` (default Vite port).

If your backend runs somewhere other than `localhost:5000`, set `VITE_API_URL` in a `.env` file in `client/`.

## How failure handling works

This was the main focus of the assignment, so here's the specific chain:

1. **Empty/invalid input** — rejected before it ever reaches the model (400 response).
2. **Missing API key** — server returns a clear 500 instead of a confusing crash.
3. **Request timeout** — the backend aborts the call to Groq after 25s and returns a 504 with a retry-friendly message.
4. **API/provider error** — non-200 responses from Groq are caught and turned into a generic "try again" message rather than leaking raw provider errors to the user.
5. **Empty model response** — checked explicitly before attempting to parse anything.
6. **Malformed JSON** — the model's raw output is stripped of any ```` ```json ```` code fences (smaller/free models often add these despite being told not to) before `JSON.parse`. If it still fails to parse, that's caught and surfaced as a clean error.
7. **Wrong shape** — even if the JSON parses, `validateRecipe()` checks that `title`, `baseServings`, `ingredients`, and `steps` are present and correctly typed before anything is sent to the frontend. Malformed ingredients/steps are caught individually.
8. **Non-food input** — the model is instructed to set an `error` field with a reason (e.g. "not food-related") instead of inventing a fake recipe, and the backend treats that the same as a validation failure.
9. **Stale responses** — if you submit a new request before the previous one finishes, the frontend tracks a request ID and an `AbortController`; only the latest request's result is ever applied to the UI, so a slow earlier response can't overwrite a newer one.

Every one of the above results in the same UI outcome: the user sees a specific, human-readable error message and a "Try Again" button — never a blank screen or a crash.

## Known limitations

- No streaming — the full recipe arrives at once rather than rendering as it generates.
- No persistence — refreshing the page loses your current recipe and checked steps.
- Free-tier/small models are less consistent at strictly following the JSON schema than larger paid models; the validation layer catches this, but it does mean occasional retries.
- No automated tests — failure paths were tested manually.
